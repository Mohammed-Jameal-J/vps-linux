01

What is a VPS?

Before you connect to anything, know what you're actually renting.

VPS stands for Virtual Private Server. A big physical machine sitting in a data center gets sliced into several independent virtual machines using a layer called a hypervisor. Each slice — your VPS — gets its own dedicated chunk of CPU, RAM, disk, and a public IP address. It behaves exactly like a full computer, except you reach it over the internet instead of sitting in front of it.

Physical Server
One real machine in a data center
→
Hypervisor
Slices it into isolated virtual machines
→
Your VPS
Your own CPU, RAM, disk & IP

Why not just use shared hosting?

Shared hosting puts your website on a server alongside hundreds of others with no real isolation — one noisy neighbor can slow everyone down, and you can't install custom software. A VPS gives you root access: full control to install anything, run any service (like Redis, a database, or a custom app), and configure the operating system exactly how you want.

▣

Dedicated resources

Your RAM and CPU allocation isn't shared the way it is on basic shared hosting.

⌘

Root access

You're the administrator. You can install, configure, and break things — and fix them.

⇄

Public IP address

Your server has its own address on the internet, reachable from anywhere.

⎘

Pay for what you pick

You choose the OS, the RAM/disk tier, and the region your server lives in.

Common providers

DigitalOcean, Linode (Akamai), Vultr, AWS Lightsail, Hetzner, and OVH are popular places to spin up a VPS. The setup steps in this masterclass apply to virtually all of them once you've created the server and have its IP address.

02

Linux Fundamentals

The operating system that runs almost every server on the internet.

Linux is a free, open-source operating system kernel originally written by Linus Torvalds in 1991. Unlike Windows, it's built around a philosophy of small tools that do one job well, configuration via plain text files, and a strict permission system — which is exactly why it dominates the server world.

Distributions ("distros")

Linux itself is just the kernel — the core engine. A distribution packages that kernel with system tools, a package manager, and default configuration into something usable. Ubuntu, Debian, CentOS, Fedora, and Arch are all distributions of the same underlying Linux kernel, with different philosophies and tools.

ConceptWhat it means
KernelThe core program that talks directly to hardware (CPU, memory, disk).
ShellThe command-line interpreter — bash is the most common — where you type commands.
FilesystemEverything is a file. Configuration, devices, and processes all live under /.
Package managerInstalls and updates software. Ubuntu uses apt.
Root userThe administrator account with unrestricted access — usually accessed via sudo.

Commands you'll use constantly

essential commands
# move around the filesystem
$ pwd                  # show current directory
$ ls -la                # list files, including hidden ones
$ cd /etc/nginx         # change directory

# read & edit files
$ cat file.txt          # print file contents
$ nano file.txt         # simple terminal text editor

# permissions & ownership
$ sudo whoami           # run a command as root
$ chmod 600 key.pem     # restrict file permissions

# processes
$ ps aux                # list running processes
$ systemctl status nginx# check if a service is running
Why permissions matter

Linux file permissions (read/write/execute, for owner/group/others) are the foundation of server security. A misconfigured permission — like a private key readable by everyone — is one of the most common real-world breach causes. You'll set this correctly in the SSH section.

03

Ubuntu Server Setup

Choosing and preparing the distribution you'll actually run.

Ubuntu is a Linux distribution maintained by Canonical, and it's the most widely used distro for VPS hosting — long-term support (LTS) releases get 5 years of security updates, predictable tooling, and the largest community, so almost every guide and package assumes Ubuntu by default.

Step 1 — Provision the server

When creating your VPS through any provider, you'll typically choose:

  • OS image: Ubuntu 24.04 LTS or 22.04 LTS (stick to LTS releases for servers)
  • Size/plan: 1–2GB RAM is enough to learn on; scale up later
  • Authentication: choose SSH key over password if offered — covered next section
  • Region: pick one geographically close to your users

Step 2 — First login & update

The very first thing you do on any fresh server, before installing anything else, is update the package index and upgrade existing packages:

first boot
$ sudo apt update                 # refresh package list
$ sudo apt upgrade -y             # install available updates
$ sudo timedatectl set-timezone UTC # keep server time consistent
$ sudo reboot                     # apply kernel updates if any

Step 3 — Create a non-root user

You'll log in as root initially, but running everything as root daily is risky — one mistyped command can damage the system. Create a regular user with sudo privileges instead:

create a sudo user
$ adduser yourname             # creates the user, prompts for a password
$ usermod -aG sudo yourname    # grants sudo (admin) privileges
$ su - yourname                # switch into the new user
Mental model

Think of root as the master key to the building, and your new user as a keycard with admin permission. You use the keycard daily; the master key stays locked away unless truly needed.

04

Connecting via SSH

SSH is the secure tunnel between your laptop and your VPS.

SSH (Secure Shell) is an encrypted protocol that lets you open a remote terminal on another machine. Everything you type, and everything the server sends back, travels through an encrypted channel — nobody on the network in between can read it.

Method A — Password login (simplest, weaker)

connect with a password
$ ssh yourname@your_server_ip
yourname@your_server_ip's password: ********
Welcome to Ubuntu 24.04 LTS

Method B — SSH key pair (recommended)

An SSH key pair has two parts: a private key that never leaves your laptop, and a public key that you place on the server. The server can verify your private key matches without the key ever being transmitted — this is far stronger than a password and immune to brute-force guessing.

generate a key pair (on your local machine)
$ ssh-keygen -t ed25519 -C "your_email@example.com"
Generating public/private ed25519 key pair.
Enter file in which to save the key (/home/you/.ssh/id_ed25519): ↵
Enter passphrase (empty for no passphrase): ********
copy your public key to the server
$ ssh-copy-id yourname@your_server_ip
# or manually:
$ cat ~/.ssh/id_ed25519.pub | ssh yourname@your_server_ip "mkdir -p ~/.ssh && cat >> ~/.ssh/authorized_keys"
now log in — no password asked
$ ssh yourname@your_server_ip
Welcome to Ubuntu 24.04 LTS (GNU/Linux)
Critical permission rule

Your private key file must not be readable by other users, or SSH will refuse to use it. Run chmod 600 ~/.ssh/id_ed25519 on your local machine to lock this down. Never share or upload your private key — only the .pub file ever leaves your computer.

Once keys work, disable password login

This single step blocks the most common attack vector against fresh servers: automated bots brute-forcing SSH passwords 24/7.

/etc/ssh/sshd_config
$ sudo nano /etc/ssh/sshd_config
# find and set these two lines:
PasswordAuthentication no
PermitRootLogin no

# then restart the SSH service
$ sudo systemctl restart ssh
Don't lock yourself out

Test your key-based login in a second terminal window before closing your current session and before disabling passwords. If the key login fails and you've already disabled passwords, you can be locked out entirely.

05

Hardening & Firewall

Close every door except the ones you're actively using.

A fresh VPS is reachable on every port unless you tell it otherwise. UFW (Uncomplicated Firewall) is Ubuntu's friendly front-end to the kernel's firewall (iptables/nftables) — it lets you allow or block traffic by port with one-line commands.

Set up UFW

configure the firewall
$ sudo ufw default deny incoming    # block everything by default
$ sudo ufw default allow outgoing   # let the server reach out freely
$ sudo ufw allow OpenSSH            # keep your SSH access alive!
$ sudo ufw allow 80/tcp              # HTTP
$ sudo ufw allow 443/tcp             # HTTPS
$ sudo ufw enable
Command may disrupt existing ssh connections. Proceed (y|n)? y
$ sudo ufw status verbose
PortServiceShould it be open?
22SSHYes — your access lifeline
80HTTPYes, if serving a website
443HTTPSYes, if using SSL
6379RedisNo — keep internal only (see next module)

Stop brute-force attempts: Fail2Ban

Fail2Ban watches log files for repeated failed login attempts and temporarily bans the offending IP address at the firewall level — an automatic bouncer for your server.

install fail2ban
$ sudo apt install fail2ban -y
$ sudo systemctl enable --now fail2ban
$ sudo fail2ban-client status sshd
  • SSH key login working, password login disabled
  • UFW enabled with only required ports open
  • Fail2Ban installed and watching SSH
  • Non-root sudo user created, root login disabled
  • System packages fully updated

    Installing Redis

    A fast, in-memory data store you'll use for caching, sessions, and queues.

    Redis (REmote DIctionary Server) is an in-memory key-value data store. Because data lives in RAM rather than on disk, reads and writes happen in microseconds — it's commonly used for caching database results, storing user sessions, rate-limiting, and message queues between services.

    Install Redis on Ubuntu

    install & start
    $ sudo apt update
$ sudo apt install redis-server -y
$ sudo systemctl enable redis-server   # start on boot
$ sudo systemctl status redis-server   # confirm it's running

    Test it works

    redis-cli
    $ redis-cli ping
PONG
$ redis-cli
127.0.0.1:6379> SET hello "world"
OK
127.0.0.1:6379> GET hello
"world"

    Secure Redis — this step is non-negotiable

    By default, Redis has no password and may bind to all network interfaces. An unprotected Redis instance exposed to the internet is one of the most commonly exploited misconfigurations — attackers actively scan for it.

    /etc/redis/redis.conf
    $ sudo nano /etc/redis/redis.conf

# 1. bind only to localhost — never expose Redis directly to the internet
bind 127.0.0.1 ::1

# 2. require a password
requirepass a_long_random_password_here

# 3. restart for changes to take effect
$ sudo systemctl restart redis-server

# 4. confirm the password is required
$ redis-cli
127.0.0.1:6379> PING
(error) NOAUTH Authentication required.
127.0.0.1:6379> AUTH a_long_random_password_here
OK
    Never expose port 6379 publicly

    In the firewall module, port 6379 was deliberately left closed. Your application server talks to Redis over localhost or a private network — Redis should never be reachable from the public internet directly.

    07

    SSL / TLS & Domains

    Encrypt traffic between your visitors and your server — for free.

    SSL/TLS encrypts the connection between a browser and your server, turning http:// into https://. It proves to visitors that they're really talking to your server (not an impostor), and that nobody in between can read or tamper with the traffic. Let's Encrypt is a free, automated certificate authority that makes this a five-minute task.

    Step 1 — Point a domain at your server

    Before requesting a certificate, your domain needs an A record pointing to your VPS's public IP address. This is set in your domain registrar's DNS panel:

    TypeHostValue
    A@your_server_ip
    Awwwyour_server_ip

    Step 2 — Install a web server (Nginx)

    install nginx
    $ sudo apt install nginx -y
$ sudo systemctl enable --now nginx

    Step 3 — Get a free SSL certificate with Certbot

    certbot
    $ sudo apt install certbot python3-certbot-nginx -y
$ sudo certbot --nginx -d yourdomain.com -d www.yourdomain.com
Successfully received certificate.
Certificate is saved at: /etc/letsencrypt/live/yourdomain.com/fullchain.pem
Deploying certificate...
Congratulations! You have successfully enabled HTTPS

    Certbot automatically edits your Nginx config to redirect HTTP to HTTPS and renews itself — Let's Encrypt certificates expire every 90 days, and Certbot installs a scheduled renewal job for you:

    verify auto-renewal
    $ sudo certbot renew --dry-run
Congratulations, all simulated renewals succeeded
    What "padlock secure" actually means

    The browser padlock confirms the connection is encrypted and the certificate matches the domain — it says nothing about whether the site's content or backend is trustworthy. SSL protects data in transit; it isn't a guarantee of overall security.

    Putting It Together

    How every piece you just learned fits into one request.

    Here's what happens, top to bottom, when a visitor loads your HTTPS site on a server you've hardened with everything above:

    Browser
    Requests https://yourdomain.com
    →
    UFW Firewall
    Allows port 443 only
    →
    Nginx + TLS Cert
    Decrypts & serves the request
    →
    Redis (localhost)
    Returns cached data instantly

    Meanwhile, Fail2Ban is quietly watching the SSH log in the background, and the only way you get in to manage any of it is through an SSH key — never a password.

    This is the same shape as production

    Real production stacks add load balancers, managed databases, and monitoring — but the core shape (firewall → reverse proxy with TLS → app → cache) is exactly what you just set up by hand.

    Knowledge Check

    Quick checks — no grading, just clarity.

    1. Why is an SSH key pair generally considered more secure than a password?

    Right — the server only ever sees your public key. Without the matching private key, there's nothing to brute-force, unlike a password which can be guessed by automated bots.

    2. Why should Redis's port 6379 stay closed in the firewall?

    Correct — Redis is designed to be accessed by trusted local applications, not the open internet. It should bind to localhost and require a password as a backup layer.

    3. What does an SSL/TLS certificate actually guarantee?

    Exactly — TLS secures the data in transit and confirms you're talking to the real domain. It says nothing about the safety of the content itself.
