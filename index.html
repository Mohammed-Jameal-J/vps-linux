Connecting via SSH

SSH na, unga laptop ku unga VPS ku idhukula irukற secure tunnel.

SSH (Secure Shell) ஒரு encrypted protocol — idha use panni, neenga ஒரு remote machine la terminal open pannலாம். Neenga type pannற ovvoru letterum, server anuppற ovvoru responsum, encrypted channel via than poagum — idhukkula network la யாரும் padikka mudiyathu. Real-world example: idhu courier la sealed envelope anuppறது mathiri — postman, delivery boy, edhuvum andha envelope ah open panni padikka mudiyathu, receiver mattum than open pannalam.

Method A — Password login (simple, weaker)

connect with a password
$ ssh yourname@your_server_ip
yourname@your_server_ip's password: ********
Welcome to Ubuntu 24.04 LTS

Method B — SSH key pair (recommended)

ஒரு SSH key pair la rendu parts irukum: private key — idhu eப்போதும் unga laptop la than irukum, veliya poagathu; and public key — idha neenga server la place pannuvinga. Server, unga private key match aaguthu nu verify pannidum, andha key transmit pannaமலே — idhu password kitta vida romba strong, brute-force guess panna mudiyaadhu. Real-world example: idhu fingerprint lock mathiri — unga fingerprint (private key) eppovum unga finger la than irukum; door la irukற scanner (public key) andha pattern ah recognize pannudhu, but unga actual finger ah copy panni eduthukka mudiyathu.

generate a key pair (on your local machine)
$ ssh-keygen -t ed25519 -C "your_email@example.com"
Generating public/private ed25519 key pair.
Enter file in which to save the key (/home/you/.ssh/id_ed25519): ↵
Enter passphrase (empty for no passphrase): ********
copy your public key to the server
$ ssh-copy-id yourname@your_server_ip
# or manually:
$ cat ~/.ssh/id_ed25519.pub | ssh yourname@your_server_ip "mkdir -p ~/.ssh && cat >> ~/.ssh/authorized_keys"
now log in — no password asked
$ ssh yourname@your_server_ip
Welcome to Ubuntu 24.04 LTS (GNU/Linux)
Critical permission rule

Unga private key file ah, vera yarum padikka mudiyaadha irukanum — illana SSH andha key ah use panna refuse pannidum. Local machine la chmod 600 ~/.ssh/id_ed25519 run panni idha lock pannunga. Private key ah eப்போதும் share pannaதீங்க, upload pannaadheenga — .pub file mattum than unga computer kitta irundhu veliya poagum.

Keys work aana pinnadi, password login disable pannunga

Idhu ஒரே step, fresh servers மேல் most common attack ah block pannidum — bots 24/7 automatic ah SSH passwords guess panni try pannudha irukum. Real-world example: idhu, house ku ஒரே ஒரு entrance mattum vechi, andha entrance ku fingerprint lock potru, regular key-lock ah completely remove pannறது mathiri — burglars key duplicate pannuradhu try panni waste pannanga, fingerprint copy panna mudiyaadhu:

$ sudo nano /etc/ssh/sshd_config
# find and set these two lines:
PasswordAuthentication no
PermitRootLogin no

# then restart the SSH service
$ sudo systemctl restart ssh
Unnaiye lock pannaதீங்க

Current session ah close panna munnadi, password disable pannamunnadi — ஒரு second terminal window la key-based login test pannunga. Key login fail aana, password um already disable aaginaal, neenga completely locked out aagi poagலாம் — real-world la, house key maathi, vandha new key work aagavillai nu theriya, old lock ah already remove pannirundhaa — andha situation mathiri.

Hardening & Firewall

Active ah use pannற doors mattum open ah vechu, மற்ற ella doors layum moodunga.

Fresh VPS na, neenga edhuvum solla போக்கினா, every port layum reachable ah irukum. UFW (Uncomplicated Firewall) na, Ubuntu oda friendly front-end — kernel oda real firewall (iptables/nftables) ku — idha vechi, port basis ah traffic ah allow/block panna ஒரு line command mattum podhum. Real-world example: idhu ஒரு apartment complex la security gate mathiri — main gate (port 22 — SSH) mattum residents ku open, side gates ella close ah irukum, security guard ovvoru entry ah check pannudhu.

UFW set up pannunga

$ sudo ufw default deny incoming    # block everything by default
$ sudo ufw default allow outgoing   # let the server reach out freely
$ sudo ufw allow OpenSSH            # keep your SSH access alive!
$ sudo ufw allow 80/tcp              # HTTP
$ sudo ufw allow 443/tcp             # HTTPS
$ sudo ufw enable

$ sudo ufw status verbose

Brute-force attempts ah nirutta: Fail2Ban

Fail2Ban log files ah watch pannudhu, repeated failed login attempts paakudhu, andha IP address ah firewall level la temporary ah ban pannidudhu — unga server ku ஒரு automatic bouncer mathiri. Real-world example: club entrance la security bouncer, ஒரே ஆள் wrong password 5 times sonna, andha aala entrance kitta வரவும் கூட விடாமல் block pannudhu — repeated suspicious attempts ah detect panni action edukudhu.

install fail2ban
$ sudo apt install fail2ban -y
$ sudo systemctl enable --now fail2ban
$ sudo fail2ban-client status sshd

  • Fail2Ban installed and watching SSH
  • Non-root sudo user created, root login disabled
  • System packages fully updated

    Installing Redis

    Caching, sessions, queues ku use pannற fast, in-memory data store.

    Redis secure pannunga — idhu skip panna mudiyaadhu

    Default ah, Redis ku password edhuvum illa, and ella network interfaces ku bind aagalam. Unprotected Redis instance, internet ku expose aaginaal — idhu most commonly exploited misconfiguration; attackers active ah idha scan pannudhu. Real-world example: idhu unga locker room la, locker ku password lock vekkamal, door ah open ah vechirukudhu mathiri — security guard (firewall) irundhaalum, locker room kulla pona yarum, edhuvum எடுத்துக்கலாம்.

    /etc/redis/redis.conf
    $ sudo nano /etc/redis/redis.conf

bind 127.0.0.1 ::1

# 2. require a password
requirepass a_long_random_password_here

# 3. restart for changes to take effect
$ sudo systemctl restart redis-server

# 4. confirm the password is required
$ redis-cli
127.0.0.1:6379> PING
(error) NOAUTH Authentication required.
127.0.0.1:6379> AUTH a_long_random_password_here
OK
    Port 6379 ah public ah expose pannaதீங்க

    Firewall module la, port 6379 ah intentionally close ah vechom. Unga application server, Redis kooda localhost via or private network via than pesanum — Redis eப்போதும் public internet la irundhu directly reachable ah irukka koodadhu.

    SSL / TLS & Domains

    Visitors ku unga server ku idhukula traffic ah free ah encrypt pannunga.

    SSL/TLS, browser ku unga server ku idhukula connection ah encrypt pannudhu, http:// ah https:// ah maathividudhu.     A@your_server_ip
    Awwwyour_server_ip

    Step 2 — Web server install pannunga (Nginx)

$ sudo systemctl enable --now nginx

    Step 3 — Certbot vechi free SSL certificate vaangunga

    Browser padlock confirm pannudhu — connection encrypted, certificate domain ku match aagudhu — idhu mattum than. Site content trustworthy ah irukka nu edhuvum guarantee pannathu. Real-world example: sealed envelope (SSL), edhuvum tamper pannama vandhachu nu confirm pannidudhu — but andha letter kulla irukra content correct ah, truthful ah irukka nu seal sollaathu.     Domain → VPS Setup

    Unga domain (xyz.com) ah VPS kooda eppadi connect pannanum — full worked example.

    Domain oda VPS ku ஒரு direct connection illa — DNS thaan andha bridge. Neenga xyz.com nu browser la type panna, andha request first ஒரு DNS resolver kitta poyi, "xyz.com oda IP address enna?" nu kekkum. Andha answer than unga VPS IP — appo dhan browser nera unga server kitta connect aagum.

    Browser
    Types xyz.com
    →
    DNS Lookup
    "xyz.com oda IP enna?"
    →
    Registrar's DNS
    Returns your VPS IP
    →
    Your VPS
    Nginx serves the site

    Step 1 — Domain vaanga (oru registrar la)

    Namecheap, GoDaddy, Google Domains, Hostinger — edhu venalum vaangalam. xyz.com nu vaangina udane, andha registrar oda dashboard la "DNS Management" / "DNS Zone Editor" nu oru section irukum — adhula thaan namma records add pannanum.

    Step 2 — A record point pannunga unga VPS IP ku

    Registrar dashboard la, இந்த records add pannunga (your_server_ip ah unga VPS oda real public IP vechu replace pannunga):

    TypeHostValueTTL
    A@your_server_ipAuto / 3600
    Awwwyour_server_ipAuto / 3600

    @ nu vechathu root domain ku (xyz.com), www nu vechathu (www.xyz.com) ku. Rendu um same IP ku point pannina, both versions work aagum.

    DNS propagation time

    Records add panna udane work aagathu — DNS changes worldwide la propagate aaga 5 minutes to 48 hours aagalam (usually 15–30 mins). nslookup xyz.com or dnschecker.org use panni check pannalam edhuvum propagate aacha nu.

    Step 3 — Server side la confirm pannunga

    check DNS propagation
    $ nslookup xyz.com
Name:    xyz.com
Address: your_server_ip   ← idhu match aagi irukanum

$ ping xyz.com
PING xyz.com (your_server_ip): 56 data bytes

    Step 4 — Nginx la domain configure pannunga

    Module 07 la nginx install pannirukom. Ippo andha domain ku specific ஒரு server block (virtual host) create pannanum:

    /etc/nginx/sites-available/xyz.com
    $ sudo nano /etc/nginx/sites-available/xyz.com
    server {
    listen 80;
    server_name xyz.com www.xyz.com;

    root /var/www/xyz.com;
    index index.html;

    location / {
        try_files $uri $uri/ =404;
    }
}
    $ sudo ln -s /etc/nginx/sites-available/xyz.com /etc/nginx/sites-enabled/
$ sudo nginx -t              # test config for errors
$ sudo systemctl reload nginx

    Step 5 — SSL kuduthu https://xyz.com aakkunga

    Module 07 oda certbot command ah ippo real domain kooda run pannunga:

    get the certificate
    $ sudo certbot --nginx -d xyz.com -d www.xyz.com
Congratulations! You have successfully enabled HTTPS on
https://xyz.com and https://www.xyz.com
    Summary — oru line la

    Domain registrar la A record → VPS IP ku point pannunga, Nginx la andha domain ku server block create pannunga, Certbot run pannunga SSL ku. Adhu thaan whole flow.

    Knowledge Check

    Quick checks — no grading, just clarity.

    1. Why is an SSH key pair generally considered more secure than a password?

    Right — the server only ever sees your public key. Without the matching private key, there's nothing to brute-force, unlike a password which can be guessed by automated bots.

    2. Why should Redis's port 6379 stay closed in the firewall?

    Correct — Redis is designed to be accessed by trusted local applications, not the open internet. It should bind to localhost and require a password as a backup layer.

    3. What does an SSL/TLS certificate actually guarantee?

    Exactly — TLS secures the data in transit and confirms you're talking to the real domain. It says nothing about the safety of the content itself.
